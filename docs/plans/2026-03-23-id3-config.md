# Spec: ID3 / Metadata User Config

**Date**: 2026-03-23 **Status**: Approved **Author**: Claude + user **Slug**: id3-config

## Plan

### Context

YD downloads YouTube audio via yt-dlp + ffmpeg. Currently no metadata, thumbnail, or title is embedded into downloaded files — yt-dlp just extracts audio and recodes it. The output filename uses `%(title)s` but no ID3 tags are written.

Item from [plan.md](./plan.md): `add user config of: title, thumbnail, metadata, HE flip`.

Prior plan [2026-03-22-yd-feature-roadmap.md](./2026-03-22-yd-feature-roadmap.md) explicitly listed this as **Out of scope**.

Reference implementation: `c:/dj/src-tauri/src/hebrew.rs` — `reverse_hebrew()` function.

### Goal

Add 3 boolean toggles to Settings so users control what gets embedded into downloaded audio: title (ID3/m4a title tag), thumbnail (cover art), and HE flip (reverse Hebrew char runs in embedded title so it displays correctly on LTR-only media players like car stereos). Filename stays unchanged.

### Approach

- **`embed_thumbnail`**: Pass `--embed-thumbnail` to yt-dlp. ffmpeg (already bundled) handles all formats natively.
- **`embed_title`**: After yt-dlp finishes, extract title from the output filename stem, then write it into the file's title tag via ffmpeg: `ffmpeg -i file -metadata title="..." -codec copy temp && rename`. No `--embed-metadata` flag needed — avoids writing unwanted extra metadata (artist, date, description).
- **`flip_hebrew_in_title`**: Same ffmpeg post-processing step as `embed_title`, but applies `reverse_hebrew()` to the title string before writing. Only active when `embed_title` is also on (no title to flip otherwise). Ported from `c:/dj/src-tauri/src/hebrew.rs`.
  - Flow: yt-dlp finishes → read title from filename stem → `reverse_hebrew(title)` → ffmpeg `-metadata title="flipped" -codec copy`
- **Config**: Add 3 bools to `AppConfig`, default all `false`.
- **No new dependencies**: ffmpeg already bundled, used for codec copy + metadata write only (instant, no re-encode).

### Alternatives Considered

| Option | Pros | Cons | Why rejected |
|---|---|---|---|
| `--embed-metadata` flag | No post-processing | Writes unwanted extra metadata (artist, date, desc), no control over title content | Can't flip HE, writes too much |
| lofty crate (like c:/dj) | Fine-grained tag control | ~15 new crate deps for writing 1 field | Overkill, ffmpeg already available |
| `--parse-metadata` in yt-dlp | Single pass | Can't reverse Hebrew runs in regex, title unknown before yt-dlp runs | Impossible |

### Scope

**In:**
- 3 new config bools: `embed_title`, `embed_thumbnail`, `flip_hebrew_in_title`
- `--embed-thumbnail` flag conditionally passed to yt-dlp
- ffmpeg post-processing to write title tag (with optional HE flip)
- `reverse_hebrew()` + `contains_hebrew()` Rust functions
- Settings UI toggles + i18n keys

**Out:**
- Filename changes (stays as `%(title)s`)
- Full metadata embedding (artist, album, date, description)
- Per-download toggle (global config only)
- Album art from sources other than YouTube thumbnail

### Risks

- **ffmpeg codec copy + metadata**: `ffmpeg -codec copy` remuxes without re-encode — instant and lossless. Requires temp file + rename. Mitigation: temp file in same directory, atomic rename on success, clean up on failure.
- **`--embed-thumbnail` + m4a**: yt-dlp uses ffmpeg for thumbnail embedding. Should work since ffmpeg is bundled. Test all 4 formats.
- **HE flip edge cases**: Mixed Hebrew/English/numbers, nikud (vowel marks). Mitigation: only reverse chars in `\u0590-\u05FF` range, leave everything else in place (proven logic from c:/dj).
- **Filename stem parsing**: Title may contain dots. Mitigation: strip only the final `.ext` to get the stem.
